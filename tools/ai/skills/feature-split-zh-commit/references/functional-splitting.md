# 功能拆分准则

## 划分顺序

1. 先按目标拆分，再按文件拆分。
2. 一个 commit 只表达一个明确目的。
3. 先隔离已有脏改动，再处理当前任务改动。

## 合并进同一个 commit 的情况

- 同一个用户可感知功能的代码与测试。
- 同一个缺陷修复涉及的多目录改动。
- 为同一行为变更必须同步调整的文档或配置。
- 同一轮重构中不可拆开的接口与实现更新。

## 必须拆开的情况

- 新功能和缺陷修复混在一起。
- 业务改动和纯重构混在一起。
- 机械性整理、格式化、重命名与行为变更混在一起。
- 构建系统调整与功能实现没有直接依赖关系。
- 当前任务改动与工作区里已有的其他改动无关。

## 脏工作区处理流程

1. 先看 `git status --short`，记录哪些改动早于当前任务。
2. 用 `git diff` 判断每个文件里的改动是否属于当前功能单元。
3. 只暂存当前单元涉及的文件或 hunk，不要顺手带上无关修改。
4. 提交前执行 `git diff --cached`，确认暂存区只有一个主题。
5. 提交后再次看 `git status --short`，再开始下一个功能单元。

## 拆分判断问题

在准备提交前，依次确认：

- 这组改动是否只回答一个“为什么要改”？
- 如果回滚这个 commit，是否只回滚一个功能意图？
- 测试和文档是否直接服务于这组改动？
- 提交说明能否用一句简短中文准确概括？

只要其中任意一项回答为“不能”，就继续拆分。
