# Kernel 风格注释工具

## 目的

本工具用于指导 AI 在 RTOS 代码中补充或重写注释。

目标不是“把代码讲全”，而是只在真正需要的地方补上简洁、稳定、工程化的注释。

## 适用场景

适合加注释的位置：

1. 位编码、标志位、高位复用、长度编码。
2. 复杂数据结构的内存或磁盘布局。
3. 多阶段流程中当前阶段的职责。
4. 旧资源与新资源的切换顺序。
5. 事务、回滚、释放、引用计数、校验和覆盖范围。
6. 二分、分层、重建树、批量分配等算法中的关键意图。
7. 看似奇怪但其实是为了正确性、兼容性或性能的写法。

不适合加注释的位置：

1. 简单赋值。
2. 直白的边界判断。
3. 常规循环本身。
4. 从函数名即可看懂的动作。
5. 与代码逐字同义的解释。

## 写法

优先使用短块注释：

```c
/* 磁盘上的长度字段使用最高位标记未写入区间。 */

/* 记录旧的元数据块，待新树写稳后统一释放。 */
```

也可以使用很短的多行块注释，但只在单行放不下时使用：

```c
/*
 * 内存视图由有序区间数组和紧凑搜索树组成。
 * 每次修改后重建搜索树，而不是做原地更新。
 */
```

避免使用：

1. `//` 风格与文件现有风格不一致时。
2. 超过 2 到 3 行的长段解释。
3. “This function ...” 之类的函数头说明。

## 语气

1. 使用陈述句。
2. 句子要短。
3. 术语要稳定。
4. 避免感叹、反问、口语化表达。
5. 避免 `obviously`、`simply`、`just` 这类弱化词。
6. 中文句子保持短，不堆砌修饰语。

## 推荐词汇

1. `内联根节点`
2. `磁盘字段`
3. `有序区间数组`
4. `搜索树`
5. `元数据块`
6. `写稳`
7. `校验和字段不参与计算`
8. `命中区间`
9. `插入位置`
10. `规范的内存视图`

## 避免词汇

1. `this function is used to`
2. `here we`
3. `obviously`
4. `simply`
5. `just`
6. `very`
7. `easy to understand`
8. `这个函数用于`
9. `这里我们`

## 决策流程

AI 在落注释前，先按这个顺序判断：

1. 这段代码是否已经足够清楚。
2. 如果清楚，不加注释。
3. 如果不清楚，先判断难点属于状态编码、布局、生命周期、算法意图中的哪一类。
4. 只写这一类信息，不扩展到函数全面说明。
5. 注释长度优先控制在 1 行，必要时最多 2 到 3 行。
6. 如果术语必须保留英文，用中文句子包住英文术语，不单独堆英文句。

## 输出约束

1. 不改变代码行为。
2. 不为了配合注释而重构函数。
3. 不引入与仓库现有风格冲突的标点或排版。
4. 不批量注释每个分支。
5. 一段逻辑通常只加一个最有价值的注释。

## 正例

正例 1：

```c
/* 校验和字段不参与 CRC 计算输入。 */
```

正例 2：

```c
/* 小树完全保存在 inode->blocks 内。 */
```

正例 3：

```c
/*
 * 内存视图由有序区间数组和紧凑搜索树组成。
 * 每次修改后重建搜索树，而不是做原地更新。
 */
```

## 反例

反例 1：

```c
/* 设置头部魔数。 */
header->magic = to_le16 (EXT4_EXTENT_MAGIC);
```

反例 2：

```c
/*
 * 这个函数初始化 extent 头，并逐个设置字段，
 * 以保证后续 extent 树可以正常工作。
 */
```

反例 3：

```c
/* 遍历所有条目。 */
for (i = 0; i < entries; ++i)
```
